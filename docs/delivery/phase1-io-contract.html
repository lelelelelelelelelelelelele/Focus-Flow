② 协议层 = 架构边界（apply-core 的边）

边界的两半：「收什么 / 吐什么 / 保证什么」（下表）＋「绝不碰什么」（见末尾 forbidden）。

入口 op（schema 锁死，additionalProperties:false 防越权）

op必填可选字段
add_taskop zoneId titledescription, priority, deadline, deadlineType, parentId, tempId（批内句柄·建新树）
update_taskop idtitle, description, completed, priority, deadline, deadlineType, parentId（按 id 部分更新）
delete_taskop id（无）· 注意级联删整棵子树

枚举锁死 / 上限

  • priority ∈ critical | heavy | high | medium | low
  • deadlineType ∈ exact | today | tomorrow | week | none
  • op 种类 ∈ add_task | update_task | delete_task · 单批上限 OP_LIMIT=50
  • urgency 不可设（deadline 派生的显示值，types/index.ts:29）

出口：PlanResult（discriminated union）

  • {kind:'noop'} — 空 ops，合法不落地
  • {kind:'error', error:{code, message, opIndex}} — 类型化错误，fail-fast 不静默
  • {kind:'plan', actions, diff, hasDeletes, deleteCount} — action 计划 + diff 预览；删除已级联展开，deleteCount/hasDeletes 供 UI 强制确认

错误码（协议保证拦截的越界）

OP_LIMIT_EXCEEDED · UNKNOWN_TASK_ID · UNKNOWN_ZONE_ID · UNKNOWN_PARENT_ID · DUPLICATE_TEMP_ID · CYCLE · INVALID_OP

边界另一半：forbidden（这块绝不碰）

纯函数：不碰 store / fs / 网络 / 命令；不修改入参 snapshot；副作用（真调 store action + saveSnapshot）留在 wiring 层。→ 与 byok-plan-v2 §4 boundary rules 一致。

树/层级边界（原 known-gap，现已覆盖 · 见 docs/harness/test-points.md）

  • TP7 一次性建新多层树 ✅ 已支持（核心层）：add_task 加 tempId、子 op 用 parentId 引用（见上「建新树·tempId」例）。⚠ app 内真正落地还需 Phase 3 wiring（addTask 回传真 id）。
  • TP8 diff 显父名 ✅：added 带 parentLabel（已有父=名字、新父=「新建:X」），形态层显示「挂到 X 下」→ 防 LLM 静默错挂。
  • TP9 re-parent 环守护 ✅：把任务挂到自身/自身子孙下 → CYCLE（见上「成环 re-parent」例）。tempId 新父是新叶子，不参与既有环判定。
